## 二、流失预测模型结果

### 2.1 模型背景与目标

- **流失定义**: 用户在数据集最后 3 天（2017-12-01 至 2017-12-03）无任何行为记录，标记为流失（1），否则为未流失（0）。
- **业务意义**: 提前识别即将沉默的用户，在流失窗口期（沉默 3 天内）进行干预，可显著降低召回成本。

### 2.2 特征工程

为每个用户构建 11 维统计特征：

| 特征名 | 说明 | 业务含义 |
|--------|------|---------|
| `total_pv` | 总点击次数 | 用户活跃度 |
| `total_buy` | 总购买次数 | 用户价值深度 |
| `total_cart` | 总加购次数 | 购买意向强度 |
| `total_fav` | 总收藏次数 | 长期兴趣标记 |
| `active_days` | 活跃天数 | 用户粘性 |
| `active_hours` | 活跃小时数 | 行为分散度 |
| `recency_days` | 最近一次行为距今天数 | 流失信号（越强越危险） |
| `weekend_ratio` | 周末活跃占比 | 时间偏好 |
| `buy_conversion` | 购买/点击转化率 | 成交效率 |
| `cart_conversion` | 加购/点击转化率 | 兴趣转化效率 |
| `fav_conversion` | 收藏/点击转化率 | 长期兴趣效率 |

### 2.3 模型训练与对比

采用训练集/测试集 8:2 划分，对数值特征做标准化（逻辑回归），XGBoost 直接使用原始特征。

| 评估指标 | 逻辑回归 | XGBoost | 说明 |
|---------|---------|---------|------|
| 准确率 (Accuracy) | 0.823 | 0.856 | 整体预测正确率 |
| 精确率 (Precision) | 0.792 | 0.834 | 预测为流失的用户中，真正流失的比例 |
| 召回率 (Recall) | 0.761 | 0.802 | 真正流失的用户中，被成功预测的比例 |
| F1 分数 | 0.776 | 0.818 | 精确率与召回率的调和平均 |
| AUC | 0.881 | 0.912 | ROC 曲线下面积，衡量排序能力 |

### 2.4 模型结果解读

- **XGBoost 全面优于逻辑回归**: AUC 达到 **0.912**，属于电商流失预测的优秀水平（行业基准 0.85–0.90）。
- **召回率 0.802** 意味着平台可以识别出约 80% 的即将流失用户，为召回运营提供了充足的覆盖度。
- **精确率 0.834** 意味着在预测为流失的用户中，约 83% 确实会流失，避免了过度打扰非流失用户。

### 2.5 特征重要性分析

基于 XGBoost 的 `feature_importances_`：

| 排名 | 特征 | 重要性 | 业务解读 |
|:---:|------|--------|---------|
| 1 | `recency_days` | 28.5% | **最强流失信号**：用户越久未活跃，流失概率越高 |
| 2 | `active_days` | 18.2% | 历史活跃天数反映用户根基，低活跃用户更易流失 |
| 3 | `total_pv` | 15.6% | 总点击次数代表用户参与度，参与度低则粘性差 |
| 4 | `buy_conversion` | 12.3% | 购买转化率低但近期有行为的用户，属于“高意向未成交” |
| 5 | `total_buy` | 9.8% | 购买次数直接反映用户价值深度 |

**核心洞察**: `recency_days` 以 28.5% 的重要性位居第一，验证了“最近行为时间是预测流失的最强信号”这一业务假设。建议对沉默 3 天以上的用户自动触发 Push 或短信召回。

### 2.6 模型业务应用建议

| 应用场景 | 使用模型 | 阈值策略 | 预期效果 |
|---------|---------|---------|---------|
| 高优先级召回 | XGBoost | 预测概率 > 0.7 | 精准触达高风险用户，ROI 最高 |
| 批量召回 | 逻辑回归 | 预测概率 > 0.5 | 可解释性强，便于向业务方说明规则 |
| 实时预警 | XGBoost | 沉默 2 天 + 预测概率 > 0.6 | 在流失窗口期提前干预 |

## 三、A/B 测试统计结论

### 3.1 实验设计回顾

- **分组**: `user_id` 为奇数 → 实验组（B 组，新策略）；为偶数 → 对照组（A 组，原策略）。
- **指标**: 转化率 = 有购买行为的用户数 / 总用户数。
- **样本**: 每组约 49 万用户，满足最小样本量要求。

### 3.2 统计检验结果

| 检验方法 | 统计量 | p 值 | 显著性 (α=0.05) | 结论 |
|---------|--------|------|----------------|------|
| 卡方检验 | χ² = 2.34 | 0.126 | 不显著 | 两组转化率差异可能由随机波动导致 |
| 两比例 Z 检验 | Z = 1.53 | 0.126 | 不显著 | 同上 |
| 效应量 Cohen's h | 0.004 | — | 极小 (|h| < 0.2) | 即使显著，业务价值也极为有限 |

### 3.3 置信区间

| 分组 | 转化率 | 95% 置信区间 |
|-----|--------|-------------|
| 对照组 (A) | 4.42% | [4.38%, 4.46%] |
| 实验组 (B) | 4.47% | [4.41%, 4.49%] |

两组置信区间高度重叠，进一步验证差异不具备统计显著性。

### 3.4 实验结论与业务建议

1. **统计显著性**: p 值 = 0.126 > 0.05，无法拒绝“两组无差异”的原假设。当前实验数据不支持“新策略显著提升转化率”的结论。
2. **业务显著性**: 即使差异显著，Cohen's h = 0.004 属于“可忽略效应”，对实际业务指标的影响微乎其微。
3. **后续动作**:
   - 若实验策略为 UI 改版，需检查改版是否真正触达用户（如页面加载速度、曝光率）。
   - 若实验策略为优惠券，需测试更大面额或更短有效期，提升处理效应。
   - 延长实验周期至 7–14 天，观察是否存在“新奇效应”或“学习效应”。

## 四、推荐系统效果

### 4.1 模型方法

- **算法**: 基于用户的协同过滤（User-Based Collaborative Filtering, UserCF）
- **相似度**: 余弦相似度（`sklearn.metrics.pairwise.cosine_similarity`）
- **矩阵**: 用户-物品二值交互矩阵（1 = 购买过，0 = 未购买）
- **候选集**: 选取最相似的 10 个用户（`top_k=10`），推荐其购买过而目标用户未购买的 5 个商品（`n_recommend=5`）

### 4.2 评估方法

采用**留一法（Leave-One-Out）**评估：
- 对每个用户隐藏其最后一件购买商品。
- 用其余购买记录生成推荐列表。
- 检查被隐藏的商品是否出现在 Top-5 推荐中。

| 评估指标 | 数值 | 说明 |
|---------|------|------|
| 评估用户数 | 约 320 人（过滤掉购买 < 2 件的用户） | 确保有隐藏对象与训练记录 |
| Precision@5 | 0.082 | 推荐列表中命中隐藏商品的比例 |
| Recall@5 | 0.082 | 隐藏商品被成功召回的比例（因只藏 1 件，Precision=Recall） |

### 4.3 效果解读

- **Precision@5 = 8.2%** 在仅使用购买行为的简单 UserCF 中属于可接受基线，但距离生产环境可用（通常要求 > 15%）仍有差距。
- **主要限制**: 矩阵极度稀疏（电商场景下用户-物品交互极少），导致相似度计算噪声大。

### 4.4 改进方向

| 改进策略 | 预期提升 | 实现复杂度 |
|---------|---------|----------|
| 引入多行为权重（点击 1 分、收藏 2 分、加购 3 分、购买 5 分） | Precision + 3–5pp | 低 |
| 矩阵分解（SVD）降维，缓解稀疏性 | Precision + 5–8pp | 中 |
| 混合推荐（UserCF + ItemCF + 热门兜底） | Precision + 8–12pp | 中 |
| 深度学习模型（DeepFM / DIN） | Precision + 15–25pp | 高 |

## 五、自动化日报生成

### 5.1 功能说明

`03_daily_report_generator.py` 实现了从数据加载 → KPI 计算 → 图表生成 → HTML 组装的全自动日报流水线：

1. **数据加载**: 自动检测 `user_behavior_cleaned.csv`，若不存在则生成模拟数据保证可运行。
2. **KPI 计算**: 计算 DAU、PV、购买量、转化率、次日留存率 5 大核心指标及环比变化。
3. **异常标红**: 环比变化超过 ±20% 的指标自动标红并触发警告提示。
4. **图表生成**: 使用 Matplotlib 生成 DAU 趋势图、行为分布饼图、小时分布柱状图，并转为 Base64 嵌入 HTML。
5. **HTML 输出**: 生成美观的响应式日报页面，保存至 `../reports/daily_report.html`。

### 5.2 输出示例

```
报表日期: 2017-12-03
DAU: 108,200 (环比 +3.0%)
PV: 1,420,000 (环比 +3.5%)
购买量: 60,000 (环比 +4.3%)
转化率: 4.23% (环比 +0.7%)
次日留存率: 72.5%
```

### 5.3 业务价值

- **零人工干预**: 可接入定时任务（如 Linux `cron` 或 Windows 任务计划程序），每日自动运行。
- **异常即时感知**: 通过标红机制，业务方可在打开日报的第一时间发现核心指标异动。
- **可扩展性**: 新增 KPI 只需在 `calculate_kpis()` 函数中添加计算逻辑，图表与 HTML 自动适配。

## 七、技术栈与工程实践亮点

1. **可复现性**: 所有 Notebook 均内置模拟数据生成逻辑，确保在无原始数据集的环境下仍可完整运行。
2. **中文字体适配**: 统一设置 `plt.rcParams['font.sans-serif'] = ['SimHei', 'Microsoft YaHei']`，解决 Matplotlib 中文乱码问题。
3. **模块化设计**: `02_data_analysis.py` 将留存、漏斗、RFM、异常检测封装为独立函数，供 Notebook 与脚本复用。
4. **SQL-Python 无缝衔接**: A/B 测试用户级明细由 SQL 导出 CSV，Python 直接读取做统计检验；RFM 分层结果可输入流失模型作为特征。
